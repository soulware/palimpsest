# Elide Status — 2026-04-09

Progress since [2026-03-30](status-2026-03-30.md).

---

## Major changes

### Volume ownership model

The volume now owns `index/` and `cache/` directly.  The `segments/`
directory is gone — replaced by `index/<ulid>.idx` (metadata) and
`cache/<ulid>.body` (evictable body data).  The coordinator writes to
`pending/` and communicates promotions via IPC; it never touches the
volume's local state directly.  This closes the evict-after-GC corruption
bug where old segments could survive and poison reads.

### Dedup materialisation (thin-local / fat-S3)

Segments written locally contain thin dedup REF entries (hash only, no body
bytes).  Before S3 upload, `materialise_segment` rewrites the segment with
full body bytes ("fat" form).  This means local storage benefits from dedup
savings while S3 segments are self-contained — no cross-segment dependency
at fetch time.  An upload sanity check rejects any thin REF before upload.

### Extent-granular demand-fetch

Replaced full-segment download on cache miss with per-extent range-GETs.
Present-bitset tracking, scan-ahead coalescing, and configurable batch sizes.
GC also uses range-GETs for live bodies instead of fetching full segments.

### Volume directory layout

Volumes use a flat `by_id/<ulid>/` layout with `by_name/<name>` symlinks
(devicemapper model).  Fork directories unified under `forks/<name>/`.
`volume.toml` replaces the scattered config files (`volume.name`,
`volume.size`, etc.).

### CLI and coordinator lifecycle

- `elide volume up/down` for starting and stopping volumes.
- `elide volume list` defaults to writable volumes; `--readonly`, `--all` flags.
- Import redesigned: synchronous by default with `--detach` and `--fork` options.
- Volume-daemon embeds coordinator tasks directly (no separate coordinator process needed for single-volume use).
- Foreground Ctrl-C shutdown with child process cleanup.
- IPC over Unix socket for coordinator communication.

### Zero extents

`NBD_CMD_TRIM` and `NBD_CMD_WRITE_ZEROES` implemented.  Zero extents stored
as a flag in the index (no body bytes), handled correctly through GC and
compaction.

### Unified index entry format

Removed `MaterializedRef` as a separate entry kind.  All entries use a
single format with `EntryKind` enum dispatch.  `FLAG_MATERIALIZED` is a
modifier on the base entry, not a separate type.

### Compression switch: zstd to LZ4

Body extent compression switched from zstd to lz4_flex.

### Segment fd CLOCK LRU cache

Replaced the single-entry segment fd cache with a CLOCK-based LRU.

### Extent index: lowest-ULID-wins

Reversed the canonical direction of the extent index so the oldest segment
providing a hash wins.  This makes dedup references stable across GC — the
target doesn't move when newer segments are compacted.

## Bug fixes

Notable correctness bugs found and fixed (most via proptest):

- **Promote must update extent index to `BodySource::Cached`** — without
  this, reads after promote could miss local data.
- **Materialise must preserve LBA-dead but hash-alive entries** — zeroing
  their bodies broke dedup references from other LBAs.
- **GC lba_live check for DATA entries** — GC was keeping entries alive by
  hash even when no LBA pointed to them, causing data loss on crash.
- **Repack must delete fully-dead segment files** — leftover files caused
  materialise to panic on missing data.
- **Evict file_cache after promote** — stale fd cache entries returned old
  data after promotion.

## Proptest improvements

- `DrainWithMaterialise` replaces `DrainLocal` as the default drain
  simulation — exercises the full materialise path.
- `DrainInterleavedWrite` tests writes concurrent with drain.
- `DedupWrite` exercises the dedup REF path in simulation.
- Deterministic regression tests materialised for every proptest-discovered
  bug (Bugs A through H).
- Coordinator GC proptests gated behind `proptest-gc` feature flag.

## Remaining work (updated)

Items 1 (`.applied` → `.done`) and 2 (segment eviction) from the previous
status are resolved by the ownership model change.  Extent-granular
demand-fetch (item 3) is now implemented.

The remaining priorities are:

1. **Delta compression** — format designed, empirically validated, not yet wired into the coordinator upload path.
2. **Boot hints + S3 repacking** — depends on access-pattern telemetry.
3. **ublk integration** — production Linux block device target.
4. **Inline extents** — threshold still zero; low priority until fetch path is proven.
5. **Snapshot indexes** — acceleration structure for cold `Volume::open()`.
