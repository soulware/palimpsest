# Elide Status — 2026-03-30

A snapshot of implementation progress, remaining work, and deliberate deviations
from the lsvd reference.  Intended as a dated waypoint; future updates should
create a new file rather than editing this one.

## What is built and working

### Core volume engine

The full write/read/crash-recovery stack is complete and property-tested.

- **WAL** — append-only, BLAKE3-hashed records (DATA + REF), fsync durability,
  recovery on open (scan + truncate partial tail + replay).  ULID pre-assigned
  at WAL creation, not flush time — this is the invariant that makes compaction
  ordering safe without a serial event loop (see Deviations).
- **Segments** — unified local/S3 format: 96-byte signed header + varint index
  + inline section + body + delta body.  Zstd compression with 7.0-bit entropy
  gate and 1.5× ratio gate (both thresholds from lsvd, validated empirically).
  Ed25519 signature per segment.
- **LBA map** — `BTreeMap<lba, entry>` with extent split/trim/merge on insert.
  Persistent cache (binary manifest + BLAKE3 freshness guard); rebuilt from
  segment index sections on mismatch.
- **Extent index** — `HashMap<blake3::Hash, ExtentLocation>`.  Rebuilt from
  segment index sections on open; updated atomically on WAL promotion and GC
  handoff.
- **Dedup** — full BLAKE3-based local dedup on every write.  Opportunistic
  cross-volume dedup on the same host.  REF records in WAL and segments avoid
  storing duplicate bodies.
- **Fork model** — snapshot markers (ULID files), `origin` files encoding the
  parent branch point, ULID cutoffs in ancestry resolution.  Layered LBA map
  rebuild: ancestors oldest-first up to branch ULID, then live fork data on top.
- **Compaction** — `pending/` idle-arm compaction with snapshot floor
  enforcement (segments ≤ latest snapshot ULID are never touched).  Output ULID
  = `max(inputs)` to guarantee it sorts below the current WAL ULID.
- **GC handoff** — `apply_gc_handoffs()` reads the new segment's index, patches
  the extent index, renames `gc/<ulid>.pending` → `gc/<ulid>.applied`.
  Integrated into the actor idle tick.

**Test coverage:** proptest ULID monotonicity oracle, crash-recovery oracle,
actor read-your-writes + GC handoff proptest, concurrent integration test
(reader thread vs. concurrent GC coordinator thread — failure mode confirmed
then fixed).

### Actor layer

`VolumeActor` owns a `Volume` exclusively; `VolumeHandle` is the `Clone + Send`
client handle used by NBD/ublk queue threads.  Reads bypass the channel via
`ArcSwap`-published `ReadSnapshot` (no lock, no round-trip).  `flush_gen` in
the snapshot ensures handle file caches are evicted on WAL promotion.

### Coordinator

- **S3 upload** — drain-pending: read `pending/<ulid>`, PUT to S3, rename to
  `segments/<ulid>`.  Uploads public key before segments.
- **GC** — two strategies per fork on a 5-minute interval: (1) density pass
  (least-dense segment below 70% threshold), (2) small-segment sweep (segments
  < 8 MB, up to 32 MB total live bytes).  Output ULID = `max(inputs).increment()`.
  Writes `gc/<ulid>.pending` before deleting inputs; awaits `.applied` before
  deleting old S3 objects.
- **Fork discovery** — inotify-based scan of configured roots; spawns
  drain-loop and gc-loop per fork.
- **Volume supervision** — spawn `elide serve-volume`, restart on exit,
  re-adopt running processes on coordinator restart.  Volumes are `setsid`
  detached (survive coordinator restart/upgrade).
- **Prefetch-indexes** — range-GET header+index sections for all ancestor
  segments; written to `fetched/<ulid>.idx`.  Enables `Volume::open()` to
  rebuild the LBA map without full segment bodies (cold-start optimisation).

### Demand-fetch

S3 or local-filesystem fetcher integrated into the read path.  On segment miss:
download full body into three-file cached format (`.idx`, `.body`, `.present`
bitset).  The three-file format is designed for extent-granular byte-range GETs;
the granular logic is deferred (see Remaining work).

### Tools

- `elide serve-volume` — NBD server (dev/test; ublk is the production target)
- `elide-import` — full OCI→volume pipeline
- `elide snapshot-volume`, `fork-volume`
- `elide inspect-segment`, `inspect-wal`
- `elide extents`, `cold-boot`, `sparse-analysis`, `rename-analysis`, `ls-volume`

## Remaining work

Roughly in priority order.

### 1. Coordinator `.applied` → `.done` (S3 space leak)

The coordinator writes `gc/<ulid>.pending` and the volume renames it to
`.applied`.  The coordinator needs to poll for `.applied`, delete the old S3
objects, and rename to `.done`.  Without this, GC reclaims local disk space but
old S3 objects accumulate indefinitely.  The filesystem side is fully designed;
only the coordinator poll + S3 delete is missing.

### 2. Segment eviction

`segments/` grows unbounded — it is designed as an evictable cache of S3
objects but currently has no size cap or eviction logic.  A simple LRU by mtime
with a configurable size cap is sufficient.  The three-file fetched format
supports partial eviction (evict `.body` but keep `.idx`).

### 3. Extent-granular demand-fetch

Currently fetches the full segment body on S3 miss.  The three-file format
(`.present` bitset, body-relative offsets) is designed for byte-range GETs per
extent.  This is blocked on eviction being in place first (full-body downloads
without eviction exhaust local disk).  Once working, this will be better than
lsvd's fixed 1 MB RangeCache chunks: fetches only what is actually read.

### 4. Delta compression

Format fully designed (zstd with source extent as dictionary; delta body
appended to segment at S3 upload time).  Empirically compelling: 94% marginal
S3 fetch saving for point-release Ubuntu updates, 86% cross-major.  Coordinator
computes deltas at upload time; volume read path already handles the format
(delta entries in the index section).  Needs coordinator-side compute and
integration test.

### 5. Boot hints + S3 repacking

Observe which extents are fetched during VM boot, accumulate per-image access
patterns, and repack those extents contiguously in S3 to reduce cold-start
fetch count.  Depends on eviction and granular fetch being in place first.

### 6. ublk integration

NBD is the current development and test tool.  Production on Linux should use
ublk (io_uring-based, zero-copy, no kernel module).  The actor layer is designed
for this: `VolumeHandle` is `Clone + Send`, one clone per queue thread.

### 7. Inline extents

`INLINE_THRESHOLD = 0` — format is implemented, code path exists, threshold
disables it.  Small extents stored inline in the index section avoid a separate
S3 fetch.  Low priority until extent-granular fetching is working.

### 8. Snapshot indexes

Consolidate all index sections at snapshot time into a single file for faster
`Volume::open()` on cold restarts.  The ground truth remains the segment files;
this is an optional acceleration structure.

### 9. Metadata-aware dedup

ext4 journal writes generate high-churn, low-value data that degrades dedup
ratios and wastes GC cycles.  Parsing committed journal transactions would let
the volume either skip journal extents or delay their dedup decision.
Significant complexity; deferred.

## Deliberate deviations from lsvd

### ULID pre-assignment (vs. lsvd's at-flush assignment)

lsvd assigns sequence numbers at flush time and uses a serial event loop to
prevent a compaction output from overwriting a concurrent WAL flush segment.
We pre-assign the ULID at WAL creation.  The invariant this buys: all
`pending/` and `segments/` files were created before the current WAL was opened,
so `max(inputs)` for compaction is always strictly less than the WAL ULID.  No
serialization needed.

This was validated by proptest: the lsvd approach fails the crash-recovery
oracle if the serial event loop is relaxed.  Our approach passes without it.
The pre-assignment also makes crash recovery detection clean: a `pending/<ulid>`
alongside a `wal/<ulid>` unambiguously means the WAL was promoted.

### Full dedup (vs. lsvd's placeholder)

lsvd has checksum infrastructure (SHA-256, base58) but no extent index and no
dedup lookup during writes.  We have a complete BLAKE3-based extent index with
local dedup on every write, including cross-volume opportunistic dedup on the
same host.  Dedup is a core design goal, not an afterthought: the two-level
indirection (LBA map → hash, extent index → location) is the reason GC can
repack without touching the LBA map.

### Coordinator as separate process (vs. lsvd's in-process event loop)

lsvd handles GC and segment closure in a background goroutine within the volume
process.  We have a separate coordinator daemon.  This enables:

1. **S3 credential separation** — volume has read-only S3 creds; all S3
   mutations (upload, delete, repacking) go through the coordinator.  Limits
   blast radius if a host is compromised.
2. **Multi-fork supervision** — one coordinator manages all forks on a host,
   with independent drain-loop and gc-loop per fork.
3. **Independent restarts** — coordinator can restart or upgrade without
   interrupting serving volumes.
4. **Clear boundary** — the coordinator never touches `pending/`; the volume
   never touches `segments/` directly after drain.  This is the invariant that
   keeps the ULID ordering safe across the coordinator/volume split.

### GC handoff protocol (vs. lsvd's in-process atomic patching)

lsvd does in-process atomic LBA map patching: hold the lock briefly, update
`segIdx` and `offset` for each copied extent, release.  This works when GC and
the volume share memory.

We use a filesystem-based handoff: coordinator writes `gc/<ulid>.pending`,
volume applies the patch and renames to `.applied`, coordinator sees `.applied`
and cleans up.  This is more crash-safe (any crash leaves the filesystem in a
recoverable state) and fits the share-nothing design (coordinator and volume are
separate processes that communicate only via the filesystem).

### Ed25519 signing (lsvd has none)

Every segment is signed with a per-fork Ed25519 key.  Signatures are verified
when fetching segments from S3, preventing a compromised coordinator or storage
layer from injecting data into a volume.  Keys are stored locally (never
uploaded); public keys are uploaded to S3 for ancestor verification.

### Compression: zstd vs. LZ4

lsvd uses LZ4 everywhere.  We use zstd (level 1 locally; level varies for delta
bodies).  Same entropy threshold (7.0 bits/byte) and ratio gate (1.5×).  zstd
at level 1 is competitive with LZ4 in speed and meaningfully better in ratio
for typical VM workloads.

### Demand-fetch strategy (currently worse; designed to be better)

lsvd fetches 1 MB chunks via RangeCache (mmap'd, LRU).  We currently fetch the
full segment body on S3 miss — worse than lsvd for cold starts.  The three-file
format (`.idx` header+index, `.body` sparse, `.present` bitset) is designed for
extent-granular byte-range GETs per read miss, which will be better than lsvd's
fixed chunks once implemented.  This is the most significant functional gap for
production cold-start performance.

## Empirical context

Measurements on a 2.1 GB Ubuntu 22.04 cloud image:

- 93.9% of image blocks never accessed during a full systemd boot (130 MB read)
- 84% of extents match exactly between 22.04 point releases (by count)
- 94% marginal S3 fetch saving for point-release update (dedup + delta combined)
- 86% marginal saving for cross-major update (22.04 → 24.04)
- ~2.7× write amplification from ext4 journal churn
- ~1.5 MB LBA map manifest for a 762 MB filesystem
