# Elide Status — 2026-04-27

Progress since [2026-04-20](status-2026-04-20.md).

---

## Major changes

### ublk transport — production target

The host-local ublk path went from speculative spike to default-enabled
production transport in this window.  Progression by step:

- **Step 1.** Initial single-queue depth-1 spike behind a `ublk` cargo
  feature; design doc landed alongside.
- **Step 2a.** Multi-queue depth-1 with lifecycle cleanup and a
  diagnostic CLI.
- **Step 2b.** Async depth-64 served via an eventfd-bridged worker pool
  to bridge libublk's blocking model with the actor's async surface.
- **Step 3.** `USER_RECOVERY_REISSUE` recovery path with sysfs-driven
  routing between fresh `add` and `recover` flows; volume↔device binding
  persisted under the volume so cross-volume recovery cannot occur.
  Survives ^C during `start_user_recover` and recovers a `DEAD` device
  cleanly.

Configuration and lifecycle:

- First-class `[ublk]` section in `volume.toml`; coordinator supervisor
  brings ublk up alongside NBD; CLI surfaces ublk state and privilege
  failures at the mutation entry point rather than on first I/O.
- `ublk` feature is now enabled by default in the workspace build.

**Shutdown-park.**  The shutdown path went through several iterations.
Final shape: leave the device in `QUIESCED` rather than calling
`STOP_DEV`/`del_dev` during shutdown, and sweep stale parked devices on
coordinator startup.  Skips redundant `del_dev` in the ADD shutdown
path.  A dedicated signalfd watcher replaces the ctrlc handler so
shutdown is deterministic instead of racing with libublk's worker
threads.  `del_dev` itself is bounded by a watchdog that escalates to
`SIGKILL` on timeout, and shutdown `del_dev` calls now use a fresh
`new_simple` ctrl to avoid the for-add-ctrls deadlock.  Build-time
errors preserve errno through to the supervisor and reclassify the
common transient ones.  `ublk_sweep` calls libublk directly instead of
shelling out to the workspace CLI.

### CI kernel lane — unconditional, blocking

`ci-kernel.yml` runs nested KVM under GitHub Actions.  Iterated through
qemu flag conflicts, in-guest module installation, 9p-mounted source,
artifact permissions, and host-side test-binary builds with in-guest
execution.  Now caches the Ubuntu cloud image and cargo registry, runs
on every PR plus `main` pushes, and is **unconditional and blocking** —
nbd::/ublk:: tests gate merges.

Coverage in the lane:

- ublk SIGKILL + SIGTERM crash-injection tests (verify recovery
  semantics, including parked-device sweep).
- Coordinator nbd lifecycle integration test (verifies that `volume
  start` followed by clean shutdown leaves no stale state).
- Minimal ublk unit tests.

A separate host-runner workflow unifies proptest features and splits
proptest into a parallel job for faster feedback.

### Reclamation — coordinator-wired

The alias-merge primitive's last open piece landed: it's wired into the
coordinator drain tick (`elide-coordinator/src/tasks.rs`) alongside
sweep / repack / delta_repack_post_snapshot, capped at one candidate
per tick.  Sustained bloat converges across ticks because the scanner
sorts most-wasteful-first.

Output-shape work also landed:

- **Snapshot-pin Delta preference.**  When the reclaimed hash H lives
  in a snapshot-floor segment, the rewritten sub-range is emitted as
  thin `Delta { source_hash: H }` instead of fresh Data — H's body is
  pinned by the snapshot anyway, so a few-hundred-byte zstd-dict patch
  beats a fresh lz4 body.
- **Preserve-delta on rewrite.**  Reclaiming a Delta-bodied entry
  preserves the delta shape: decompress the fragment, slice the live
  sub-range, re-compress against the same source, and emit a thin
  Delta with one option for that source.  Net-loss deltas fall back to
  skip; sources that don't resolve locally are skipped (no
  demand-fetch).
- **Bloat gate aligned with scanner.**  Both the scanner and the
  primitive use `live_blocks < logical_blocks` so tail-overwrite
  candidates are no longer silently rejected.

Phase 2 of the primitive runs on the worker thread; the actor sees
prepare and apply only.  WAL flush ordering invariant `u_flush <
u_reclaim` is enforced — without it, a flushed WAL entry could outlive
the reclaim output and produce zero reads on rebuild.

### Real S3 credentials, coordinator-vended

The volume / coordinator credential boundary became real:

- `S3 credential backends` design doc lays out the issuer, refresh
  strategy, and a Tigris quickstart.
- Coordinator now exports S3 store config to volume subprocesses on
  spawn; volumes never read store config from disk directly.
- `volume create --from <vol/snap>` works against a real remote — fork
  + extents-from reads no longer trip on missing local state.
- **Macaroon-authenticated credential vending.**  The volume requests
  S3 credentials over the coordinator IPC, presenting a
  volume-specific macaroon.  The coordinator validates the macaroon
  against its root key, vends scoped credentials, and refreshes them
  on expiry.
- Coordinator creates `data_dir` before loading the root key (was a
  first-boot crash).

### Coordinator config + supervisor section

`coordinator init` writes a TOML template; durations parse as
humantime (`24h`, `10m`).  A new `[supervisor]` section centralises
restart policy, retention windows, and tick cadence.  The default
`retention_window` dropped from 24h → 10m — short enough that idle
volumes don't accumulate retention markers indefinitely, long enough
to absorb a coordinator restart.  The template now includes a Tigris
endpoint example.

### Retention markers (renamed from pending_delete)

Cross-host retention now uses `retention/<ulid>` markers instead of
the earlier `pending_delete/`.  The marker shape is unchanged but the
naming reflects what they actually do (hold a segment for a window,
not "schedule deletion").  The replica-model doc gained a Retention
economics section explaining the cost-of-keeping vs cost-of-refetch
tradeoff.

### CLI

- **`volume fork` retired.**  `volume create --from <vol_ulid>/<snap>`
  is the sole path to a forked volume.  Unifies CoW + remote-pull into
  one command.
- **`volume create --force-snapshot`** for forking from a remote
  volume that has no owner snapshot yet.  Pairs with a stable
  attestation key on the forking host so `--force-snapshot` is
  fail-fast against missing material.
- **`volume list` rework.**  New ULID, MODE, TRANSPORT, PID columns;
  defaults to all volumes; `--all` includes pulled ancestors;
  honestly reports coordinator-down state instead of pretending.
- **CLI mutations route through coordinator IPC.**  Non-root CLI
  against a sudo coordinator now surfaces ublk privilege failures at
  the right boundary instead of bubbling raw libublk errors.

### Volume crate refactor

`elide-core/src/volume/mod.rs` was split topic-by-topic over four PRs:

- `readonly.rs` for `ReadonlyVolume` + ancestor walkers, then split
  again into `readonly` / `fork` / `wal`.
- `repack.rs` and `reclaim.rs` for sweep/repack/delta-repack and
  alias-merge data types respectively, with the `impl Volume` blocks
  moving to siblings.
- `jobs.rs` for worker-offload job/result types.
- `tests/` directory with shared `test_util.rs`, tests split by
  topic.
- Read path and compression helpers extracted from `mod.rs` (last PR
  in the window).

`VolumeHandle` was split into `VolumeClient` + `VolumeReader` to make
the read-only surface explicit and to let ublk hold a reader without
the writer rights.

### NBD perf

- `TCP_NODELAY` set on accepted connections; reply header coalesced
  into a single `writev` to avoid sending the 16-byte header in its
  own segment.
- Multiple in-flight requests per connection (was previously
  serialised at the connection layer).

### Pulled-ancestors layout

The `readonly/` tree was collapsed into `by_id/` — pulled ancestors
now live in the same directory shape as local volumes, keyed by ULID.
`volume list` learnt to enumerate them.

### Tigris-native exploration

Two design docs landed (status: exploratory):

- `design-tigris-native.md` — what Elide looks like when implemented
  directly against Tigris's snapshot + IAM primitives instead of plain
  S3.
- `replica-model.md` proposal section on using Tigris snapshots as a
  materialize read source (historical materialize).

These are exploration, not commitments — Tigris snapshot-enabled
buckets were ruled out as a snapshot mechanism (storage grows forever),
but per-key prefix-scoped IAM is a viable per-volume credential path.

## Bug fixes

Notable correctness fixes in this window:

- **Hash-verify demand-fetched bodies and delta blobs** before writing
  them to local cache.  A corrupted S3 read could otherwise persist as
  a poisoned cache entry indefinitely.
- **Bounds-check entry offsets and lengths** at segment-parse time.
  A malformed segment header could previously index off the end of the
  body section.
- **Cap segment entry count at 8192** for flush and pack — caps the
  worst-case ratio between header and body for sweep outputs.
- **`apply_gc_handoffs` serialised** to prevent dropped reply senders
  on concurrent invocation.
- **`apply_gc_handoffs` ancestor walk** in coordinator liveness
  rebuild — ancestor-provided hashes were being missed and producing
  false dead-entry classification.
- **Republish snapshot after `prepare_reclaim` flushes WAL** — the
  snapshot pinned the pre-flush state and reclaim could splice against
  it, producing zero reads on rebuild.
- **`segment_cache` keyed by signature bytes**, not file length — two
  segments of the same length but different content would collide.
- **Promote: flip `DeltaBodySource` Full→Cached on drain** — drained
  segments retained `Full` and tried to upload bodies that already
  lived in S3.
- **Fetch: reject empty range-GET** instead of panicking — happens
  for hash-only DedupRef entries.
- **Redact drops dead entries; repack invalidates promote siblings.**
  Two coupled bugs: redact left dead-LBA entries that GC then refused
  to drop, and repack didn't invalidate the cached promote sibling
  segments after rewrite.
- **WAL creation deferred to first write** to stop idle GC tick churn
  on freshly-opened volumes.
- **GC test helper emits `Canonical`** when an entry is kept only via
  hash liveness — was emitting Data and producing false positives in
  partial-death tests.
- **GC dead/repack/sweep merged into a single compaction pass** —
  three separate passes were producing redundant work and inconsistent
  observable state mid-tick.
- **`WalFlushAboveRepack` → `WalFlushAboveGcOutput`** in the TLA+ spec
  to match the merged-pass shape.
- **Idle-tick logs demoted to debug** for routine reclaim and
  dead-entry events.

## Remaining work

ublk integration (item 2 in the previous status) has gone from
"production Linux block device target" to a default-enabled transport
with crash-injection coverage and supervisor wiring; zero-copy is
explicitly deferred.  The reclamation scheduler is wired; only
threshold tuning remains, and that has the same telemetry dependency
as boot hints / S3 repacking.

Remaining priorities:

1. **Boot hints + S3 repacking** — still depends on access-pattern
   telemetry; repacking heuristic sketched in the delta doc.
2. **Snapshot indexes** — acceleration structure for cold
   `Volume::open()`.
3. **Reclamation thresholds** — `ReclaimThresholds` defaults are
   placeholders.  Picking a real "worth firing" bar (and a per-tick
   cap above 1) needs empirical work on an aged volume — same
   telemetry dependency as item 1.
4. **ublk zero-copy** — deferred until the rest of the transport
   stabilises; the design doc records the deferral.
5. **Inline extents** — threshold still 256 bytes; low priority until
   fetch-path telemetry says otherwise.
