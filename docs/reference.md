## Implementation Notes

**The NBD server is a development and testing tool.** The production block device frontend will be ublk (Linux, io_uring-based). NBD is kept for development convenience and macOS compatibility during local testing. This is architecturally identical to the lab47/lsvd reference implementation, which also exposes an NBD device. Elide's NBD server listens on TCP rather than a Unix socket purely for convenience during dev/test (e.g. connecting a VM running under Multipass or QEMU without configuring shared sockets). No design decisions should be made to optimise the NBD path at the expense of the ublk path.

**S3 upload is implemented via `elide-coordinator`.** The coordinator daemon drains `pending/` to the object store automatically on each tick, renaming each segment to `segments/` on success. Both local (`object_store::LocalFileSystem`) and real S3-compatible stores are supported with no code change.

**Fork ownership is share-nothing by design.** Each fork is fully self-contained on its host — no distributed locks, leases, or coordination between hosts. The only inter-host communication is one-directional: segments flow from local disk to S3 via the coordinator. Forking to a new host is a clean handoff via snapshot, not a live migration. Each fork has an Ed25519 keypair (`volume.key`, `volume.pub`) generated at creation; every promoted segment is signed. On `serve-volume` open, `volume.provenance` (hostname + canonical path + signature) is verified as a sanity check against accidental copies. This is not a key management system — if `volume.key` is copied to another host, both become valid signers. The guarantee is narrow: it catches misconfigured coordinators writing to the wrong fork and provides per-segment integrity at demand-fetch time. Full write access control is a bucket-level IAM concern.

**Imported readonly volumes have a public key only.** `elide-import` generates an ephemeral Ed25519 keypair in memory before writing any segments, signs all segments and the provenance file with it, writes `volume.pub` and `volume.provenance` to disk, then discards the private key. `volume.key` is never written. Since a readonly volume can never accept new writes, the private key has no use after import completes. The signing logic lives in `elide-core::signing`, shared by both `elide` and `elide-import`.

**Demand-fetch is implemented (phase 1).** Segments in `segments/` are S3-backed and evictable; segments in `pending/` are local-only and must not be evicted. The hook point is `find_segment_file()` in `elide-core/src/volume.rs` — after all local directory checks fail, it delegates to an optional `SegmentFetcher` trait object, which downloads the segment and caches it in `segments/`. The fetcher is implemented in `elide/src/fetcher.rs` using `object_store` with a `tokio::runtime::Runtime::block_on` wrapper to satisfy the sync interface.

Key design decisions:

1. **`SegmentFetcher` trait in `elide-core`** (mirrors `SegmentSigner`): sync interface, `fn fetch(&self, segment_id: &str, dest: &Path) -> io::Result<()>`. `elide-core` stays async-free; async fetcher wraps `block_on`.
2. **Eviction deferred (phase 1)**: `segments/` grows unbounded. Eviction (LRU size cap) is the natural next step once fetch is validated in production.
3. **Fetch config via `fetch.toml`** in the volume directory; env var fallback (`ELIDE_S3_BUCKET`, `AWS_ENDPOINT_URL`). A `local_path` key enables local-filesystem testing with no object store server.
4. **Ancestor public keys via coordinator upload**: the coordinator drain loop uploads `volume.pub` to `by_id/<volume-ulid>/volume.pub` before any segments, so the key is always present when segments are fetchable. Signature verification at fetch time (TOFU) is not yet wired — the segment signature is stored in the file and verified at `Volume::open`, but the demand-fetch path does not verify before writing to `cache/`.

**ULID crate: `ulid` over `ferroid` (as of March 2026).** Both crates provide monotonic ULID generation. `ferroid` v2.0 (March 2026) is more capable — lock-free `AtomicMonoUlidGenerator`, pluggable `TimeSource`/`RandSource` traits, `no_std` support, async-cooperative generators — but it hit its first major version bump after only ~10 months of existence (55 releases) and its API stability is unproven. `ulid` is mature (14M downloads, stable since 2017) and its single `Generator` type is entirely adequate for the current design, where the coordinator runs a single GC loop with its own generator and concurrent ULID allocation is not a requirement. Revisit `ferroid` once it has a few stable quarters post-2.0, or if the write path evolves toward lock-free concurrent ULID allocation.

**Compression: full-extent granularity, lz4_flex for body, zstd for deltas (delta not yet implemented).** The lsvd reference implementation uses LZ4 for all extent compression and has no delta compression. Elide matches lsvd's granularity model: `FLAG_COMPRESSED` applies to the full stored payload; decompression on read always decompresses the complete extent, even for sub-range reads. The divergence is algorithm choice. lz4_flex is used for locally-written body extents (fast decompression on the hot read path — same rationale as lsvd; benchmarked at 14–25× faster decompression than zstd level-1 on 4KB blocks). zstd is reserved for delta bodies in S3 (better ratio justifies slower decompression; delta blobs are small and fetched infrequently). lz4_flex is pure Rust (no C dependency). This two-algorithm design is an Elide-specific addition; lsvd has no delta compression at all.

**Local read path performance.** Each `Volume::read` call does one BTreeMap range scan (LBA map), then for each extent: one HashMap lookup (extent index), one file-handle cache check, and if the cache misses, O(ancestor_depth) `stat` syscalls to locate the segment plus a file `open`. The file-handle cache holds a single entry — effective for sequential reads within one segment, but any read that spans N distinct segments costs N file opens in sequence. Compressed extents are decompressed in full on every read; partial reads of a large compressed extent are O(extent_size) not O(requested_size). Both costs are bounded by the segment size limit and the pre-log coalescing block limit. Widening the file-handle cache to a small LRU (8–16 entries) is low-effort and would significantly reduce the per-read overhead for random-access workloads spanning multiple segments.

**Next: segment eviction.** `segments/` is now a cache tier but has no size cap. A simple policy: scan `segments/` by `mtime` (touched on fetch), evict oldest when total size exceeds a configured cap. Only `segments/` files are evictable — `pending/` files must never be removed (they are not yet in S3).

**Cache segment format (three-file).** On a demand-fetch cache miss, the fetcher downloads the full S3 object and writes it as three files: `index/<ulid>.idx` (header + index + inline; always written first so rebuild works on restart), `cache/<ulid>.body` (body bytes; body-relative offsets — offset 0 = first body byte), and `cache/<ulid>.present` (packed bitset; one bit per index entry; all bits set since the current fetcher always downloads the full body). The read path checks `cache/<ulid>.body` before calling the fetcher, and `extentindex::rebuild` and `lbamap::rebuild_segments` scan `index/*.idx` at startup so evicted segments remain visible in the index. The three-file format is designed for future extent-granular fetching: individual extents can be written into the sparse `.body` file at their `body_offset` and their bit set in `.present`, without any format change. A background promotion step (planned, not yet implemented) collapses a fully-populated cache triplet into a single `segments/<ulid>` file.

## Open Questions

- **Hash output size:** BLAKE3 at full 256-bit is the current choice — collision probability is negligible (~2^-128 birthday bound) at any realistic extent count, and speed is equivalent to non-cryptographic hashes on AVX2/NEON hardware. A truncated 128-bit output would halve the per-entry cost in the extent index while keeping collision probability effectively zero at practical scales. Worth revisiting once the index size and memory pressure are measured empirically.
- **Inline extent threshold:** extents below this size are stored inline in the segment's inline section rather than referenced by body offset. Needs empirical validation against the actual extent size distribution in target images.
- **Entropy threshold:** 7.0 bits used in experiments, taken from the lab47/lsvd reference implementation. Optimal value depends on workload mix.
- **Segment size:** ~32MB soft threshold, taken from the lab47/lsvd reference implementation (`FlushThreshHold = 32MB`). Not a hard maximum — a segment closes when it exceeds the threshold. Optimal value depends on S3 request cost vs read amplification tradeoff.
- **Pre-log coalescing block limit:** lsvd uses 20 blocks. The right value for Elide depends on typical write burst sizes, acceptable memory footprint between fsyncs, and worst-case read amplification when compression is enabled (see reference-lsvd.md). The limit must be enforced at the write path even when the NBD layer delivers larger contiguous writes — splitting oversized writes into capped extents is preferable to unbounded amplification.
- **LBA map cache invalidation:** validate the cached `lba.map` against a hash of the current segment IDs across the full ancestor tree, not just the live node.
- **Delta segment threshold:** not every segment needs a delta body — only useful when changed extents have known prior versions in the ancestor tree. Criteria for when to compute and upload a delta body need empirical validation.
- **Boot hint persistence:** where are hint sets stored, how are they distributed across hosts? See reference-nydus.md for how nydus approaches this (embedded in image bootstrap at build time).
- **Boot hint ordering:** record LBA access sequence with timestamps (not just a set) to enable priority-ordered prefetch — see nydus NRI optimizer elapsed-time approach.
- **Empirical validation of repacking benefit:** measure segment fetch count before and after access-pattern-driven repacking.
- **ublk integration:** Linux-only, io_uring-based. NBD is the dev/test frontend; ublk is the production target.
- **Read path file-handle cache:** the current implementation holds a single cached `File` handle. Widening to a small LRU (8–16 entries) would reduce per-read overhead for workloads that touch extents across multiple segments (VM boot, random I/O) without meaningfully increasing memory usage. Straightforward to implement; evaluate against profiling data from a real boot trace.
- **Extent-granular fetching:** the current demand-fetcher downloads the full segment body on a cache miss. The three-file cache format is already designed for per-extent byte-range GETs (`.present` bitset, body-relative offsets in `.body`). The missing piece is the fetch-path logic to issue a GET for just the needed extent, write it at the correct body offset, and set the bit — deferring the rest of the segment body. Prioritise after eviction is working and bandwidth cost is measurable.
